# READOUT GOVERNANCE AND ANTI-PATCH-ILLUSION — STAGE 1

## Why this file exists
This file records the project's governance rules for interpreting farther-scale success.
It exists to prevent the project from confusing three different things:
1. physical-response evidence
2. readout-compensator evidence
3. score-chasing patch accumulation

The project is allowed to use low-complexity readout separators.
It is **not** allowed to quietly rename readout compensation into proof that the physical body is solved.

## Core governance rule
A farther-scale pass must always be reported in **two layers**:

### Layer A — physical-response evidence
Ask:
- are responses still structured rather than random?
- is the failure mode a narrow semantic leak or a broad collapse?
- does the same class family stay clustered across seeds and profiles?
- is drift interpretable in terms of scale / profile / overview geometry?

### Layer B — readout-compensator evidence
Ask:
- was the separator selected only from seen scales?
- was the threshold fixed before target evaluation?
- were target labels used at any point in selector design?
- is the separator low-complexity?
- what exact error family does it repair?

A result is only clean if both layers are reported.

## Naming discipline
Do not present every farther-scale separator as if it were a new physical law.
Use the following naming preference:
- `separator`
- `safeguard`
- `compensator`
- `veto` only when the mechanism is literally a veto rule

Do **not** imply:
- that a separator proves the physical core is solved
- that a pass at one farther scale proves unbounded scale generalization
- that a fixed readout rule is automatically evidence of direct symbolic emission from the cell sphere

## Anti-patch-illusion rules
Do not:
- celebrate `1.000` without describing what error family was repaired
- stack narrow rules silently and then narrate the result as “theory confirmed”
- promote target-scale-specific rescue to clean generalization
- infer physical-core death from a single farther-scale failure without checking whether the failure remains structured

Do:
- preserve honest distinction between overview geometry and compensator geometry
- report whether a new rule changed threshold, sign restriction, profile restriction, or target dependence
- treat fixed seen-scale-only separators as valid evidence of structured semantics, but not as proof that compensation is unnecessary
- escalate to route change only when failures become broad, unstable, or target-dependent

## When route change is justified
A route change becomes justified only if one or more of the following happens repeatedly:
1. farther unseen scales fail even under fixed seen-scale-only low-complexity separators
2. failure modes stop being narrow and become broad semantic collapse
3. separators require target-scale labels or repeated retuning to keep working
4. overview-first no longer gives a cleaner explanation than local patch stacks

Until then, the correct stance is:
- keep the matter-first interpretation
- keep the overview-first mainline
- keep farther-scale separators explicitly labeled as separators
- continue testing broader scales and nuisance before declaring either victory or collapse



## V30 case study — the danger of overgeneralizing a single farther-scale residual
The `N224 seed 9 translation_x_neg_late_soft` residual is a useful governance example:
- it is easy to narrate as “the architecture fails beyond N192”
- but the audited evidence shows a much narrower truth: a stage-1 late-soft gate drift for one negative-translation case

Governance lesson:
- a single farther-scale miss should be localized first (gate vs sign vs separator vs target-tuned rescue dependence)
- only after that localization is it legitimate to discuss route changes such as physical-core redesign or architecture demotion


## V31 case study — when a farther-scale residual exits seen support
The `N224 seed 9 translation_x_neg_late_soft` residual has now been pushed one step farther: a boundary study checked whether it could be repaired cleanly **inside the current five-key gate family** using seen-scale-only support.

Result:
- the residual remains sign-coherent toward `translation_x_neg`
- but it exits seen `translation_x_neg_late_soft` support on all checked gate-support axes simultaneously
- therefore a clean repair **inside the current gate family** is not supported by evidence

Governance lesson:
- not every narrow farther-scale miss should trigger another threshold tweak
- if the residual has already left seen support on multiple axes, the honest next branch is a **gate-basis study** (for example sign-aware fallback or richer temporal/trace gate) rather than another narrowly narrated farther-scale veto


## V32 governance note — controlled fallback versus patch inflation
A controlled fallback may be admitted when all of the following are true:
1. it is derived from seen-scale evidence only
2. it leaves seen-scale behavior unchanged
3. it activates on a sharply delimited residual family
4. it is described explicitly as a fallback layer, not as proof that the mainline gate basis was already sufficient

The N224 sign-aware fallback currently meets these conditions on the standard seed-expansion panel. It should therefore be described as a **controlled fallback candidate**, not as hidden patch inflation and not as a new farther-scale mainline rule.

## Governance note on recoverability claims
Do not speak as if the project has already proven unrestricted all-scale semantic recoverability. Current evidence is about **controlled recoverability under audited panel constraints**. Where a residual leaves the current gate basis but remains coherent in sign or temporal structure, the correct question is whether information disappeared or whether the current projection became insufficient.


## V33 governance note — fixed fallback stress tests matter more than new patch names
When a controlled fallback survives a panel family change **without threshold changes and without activation inflation**, that is meaningful evidence about its governance status. In this case the N224 sign-aware fallback now survives both standard seed expansion and the first harder unseen-nuisance panel while remaining dormant on seen scales and on the harder target set.

Governance lesson:
- a fixed fallback that stays quiet under panel change is less suspicious than a fallback that needs continual retuning
- but survival on one harder panel is still not enough to rename it as mainline
- broadened seed coverage remains the required next step before broader promotion



## V34 governance note — sparse fallback success is not the same as sign-complete closure
A carried-forward fallback may remain well-behaved (sparse activations, no seen-scale pollution) and still fail to provide a sign-complete farther-scale account. The v34 N224 harder seed-expansion result is an example: one negative residual is rescued cleanly, while one positive residual remains. Governance consequence: report both the controlled behavior and the asymmetry; do not collapse them into a single headline such as “fallback solved farther-scale harder nuisance”.




## Sign-asymmetry governance rule
If a farther-scale fallback rescues one sign family cleanly, do **not** assume the opposite sign should receive a mirrored fallback. A mirrored fallback requires its own seen-scale support audit. Otherwise the project risks confusing a controlled recovery layer with target-informed threshold multiplication.


## Update — v36 observation failure is not information absence
For farther-scale residuals, the governance stance is now explicit:
- a current gate-basis miss should not be casually rewritten as “the information is gone”
- first separate sign failure, separator failure, gate-basis failure, and likely SNR loss
- only after that separation should the project decide whether to accept the residual, extend the basis, or stop the branch

This rule matters for the current `translation_x_pos_late_balanced` residual: the audit indicates a gate-basis miss under the present readout stack, not a proven disappearance of physical response information.


## V37 governance note — projection miss versus patch temptation
When a farther-scale residual becomes recoverable under a seen-scale-only richer temporal / trace probe that is already emitted by the current stack, prefer describing the result as a **basis extension** rather than a **new patch**. This is especially important when the probe fires sparsely, leaves seen scales untouched, and does not require target-scale threshold reselection.

## V38 governance note — where residual discussion belongs
Residual discussion should now live in a dedicated axis rather than only inside the tail of latest reports.
Use:
- `RESIDUAL_TAXONOMY_AND_RELATION_HYPOTHESES_STAGE1.md` for classification, family naming, and relation evidence
- `GEOMETRIC_CANONICALIZATION_AND_RESIDUAL_RELATION_HYPOTHESES_STAGE1.md` for appendix-level canonicalization / relation hypotheses

Governance lesson:
- recovery and residual classification are both valid outputs
- not every residual discussion should be forced into “fix immediately” or “declare impossibility” narratives
- residuals may be relation evidence even when mainline promotion is not yet justified

## v39 governance note — silent standard probe matters
A fallback or bridge that fixes a harder-panel residual but then also fires across a standard panel is governance-negative because it suggests hidden contamination pressure. The N224 positive trace-basis bridge passed an important anti-patch check: on the standard N224 seed-expansion probe it remained completely silent while preserving perfect accuracy.

Governance meaning:
- silence on standard rows is evidence that the bridge is not broadly warping the readout geometry
- this still does **not** justify immediate mainline promotion
- promotion should wait for farther unseen scale validation with the full fixed stack



## V40 governance note
A farther-scale pass becomes more trustworthy when support layers remain silent. The first beyond-N224 standard probe at N256 passes with the current full stack while both the negative sign-aware fallback and the positive trace-basis bridge remain inactive. This should be reported explicitly because it distinguishes “needed compensation” from “available but unused support.”



## V41 governance note
A clean first-pass win at a harder farther unseen scale does not automatically license stronger promotion claims. Governance discipline remains:
- keep support-layer silence/trigger counts explicit
- require seed expansion before broader farther-scale promotion
- continue to distinguish between controlled support and mainline mechanism


## V42 governance note
A seed-expansion pass is stronger than a first two-seed probe only when the support layers remain sparse or silent. The N256 harder seed-expansion pass matters precisely because both controlled support layers remain silent across the expanded target set. This should be reported explicitly to distinguish controlled geometry from hidden activation inflation.
## V43 governance note
A clean seed-expansion pass on both the standard and harder panels matters because it shows that the current full stack is not merely surviving on a narrow target slice. The key governance signal at v43 is not only `1.000 / 1.000`, but the continued silence of both support layers across the expanded N256 standard target seed set.

## V44 governance note
When a farther unseen scale first fails while all controlled support layers remain silent, the required action is **residual audit before support expansion**. This is precisely the opposite of patch-chasing: freeze the current stack, classify the residual pair, and only then decide whether the current basis is insufficient or whether a new support family is justified.


## V45 governance note — pairwise residual discipline
When the first residuals appear at a new farther scale, treat them as **potentially different morphologies** until proven otherwise. At N288 the project now has one separator-overreach case and one baseline-edge leak. A single new support rule that absorbs both without separate white-box justification would violate the anti-patch discipline.


## Residual atlas rule (v46)
Before a new support rule is proposed for a farther-scale residual, the residual must be entered into `RESIDUAL_CONDITION_ATLAS_STAGE1.md` with:
- canonical case
- residual family name
- current best condition bundle
- current support status

This rule exists to keep the project from becoming a sequence of unexplained rescue layers.


## Update — controlled support discipline for separator overreach
The N288 separator-overreach family is the first case where a farther-scale support-layer boundary itself crossed into a true-translation sample. The project responded with a **controlled study**, not with open-ended patch growth:
- one guard only
- threshold derived from seen support
- explicit re-check on previously clean farther-scale panels
- no escalation to harder nuisance until the second N288 family is also classified

This is the current template for any future support-boundary residual.

## v55 governance note
A repeated farther-scale error family created by a carried-forward guard is more urgent than an isolated spill. The correct response is pair-specific controlled study on the repeated family first, not immediate mirrored repair on every new miss.
